DevOps 與 SRE 的基礎認知

你分得清楚 DevOps 與 SRE 這兩個名詞的不同嗎

很多開發者在聊到 DevOps 或是 SRE 的時候，腦中都會關聯到技術實作，像是 Jenkins 或是 CI/CD 等等， 在聊 DevOps 以及 SRE 前，我們先來講一個故事。

軟體業的鬼故事

M Shop 是一個電商網站，M Shop 公司有一個軟體開發團隊，團隊內有 Developer 以及 Operator 這二種角色。
Developers 盡責地交付產品團隊規劃的需求，Operator 則努力維持服務穩定。

隨著業績蒸蒸日上，使用者愈來愈多。團隊開始收到使用者回報的問題，像是無法登入、無法完成下單甚至是無法結帳等。
Developers 忙著完成產品團隊規劃的功能並交付、 Bug 的修正以及線上問題的處理，Operators 面對沒有優化的系統只能透過加機器或調網路架構或系統架構等方法維持服務穩定。
Developers 對生產環境的的了解很少，做的程式優化往往不是核心問題點；Operators 對實際的程式實作了解甚少，能調整的系統優化有限，較大型的架構調整也因過度分工和缺少現有程式運作上的相關知識難以小步調整。

Developers 希望更快的交付功能，Operators 希望得到穩定的功能交付以減少服務不穩定的情形，兩個不同目標的群體導致了合作的不順暢甚至是對立。
Developers 責怪 Operators 的服務不穩定導致問題頻頻發生；Operators 責怪 Developers 的程式碼寫的太糟導致維運困難。

由於 Developers 以及 Operators 都缺乏了對 M Shop 的完整認識，導致線上問題如 bug 或系統不穩等問題處理工作的比例愈來愈高，
慢慢的，Developers 能做功能交付時間愈來愈少，功能交付愈來愈慢，工作效率也因為無力做工作流程的優化無法提升，Operators 看著不穩定的系統只能治標解決，產品團隊開始責怪整個開發團隊效率低落...。

問題在哪

上面的鬼故事身為讀者的你有沒有聽過甚至親身體驗過呢?

我們可以從上面的故事可以發現以下幾個問題:

  1. Developer 以及 Operator 由於過度分工且缺少溝通，目標的不同導致了責任的不同，這兩種角色就像是二個小公司，只關注自己職能的目標，各自為政且存在著知識流通的問題，產生了穀倉效應（The Silo Effect）。
  2. 由於互相的不理解產生了對立，互相責怪下沒有人願意說實話且合作解決問題。
  3. 由於 Developers 較專精於功能開發，Operators 較專精於通用的維運知識，對於現有程式的理解不充足。Developers 與 Operators 難以合作推進一些小步且安全的優化。
  4. Developers 與 Operators 雜務纏身導致忙於追趕工作，沒有額外力氣與心思花在優化工作效率。
  5. 你注意到上面的故事，只提到了服務穩定，卻沒有更量化的定義什麼叫做穩定嗎?

DevOps 與 SRE

DevOps 是一種哲學，不是一種開發方法或技術。
DevOps 強調開發團隊的關鍵運作方式，但卻沒有明確定義如何執行。

以下是五個 DevOps 強調的關鍵點

  1. 降低組織的穀倉
  2. 接受錯誤的發生
  3. 小步疊代
  4. 善用工具以及實作自動化
  5. 測量一切工作

而 SRE(Site Reliability Engineering) 是一種 DevOps 的技術實作及文化實踐，也是一種角色(Site Reliability Engineer)。
SRE 的任務為提供可用的系統，以及改進系統，並關注於系統的:

  • 可用性(availability)
  • 延遲(latency)
  • 效能(performance)
  • 容量(capacity)

以下會介紹 Site Reliability Engineering 是如何明確定義了 DevOps 的運作。

降低組織的穀倉

由於職能的分工，Developers 通常不明白開發完成的程式碼所執行的生產環境的全貌，而 Operators 必須處理這些可能不穩定的程式碼。
相互的不理解導致難以合作甚至是處於對立。
軟體工程專注在設計以及開發軟體，而不是維運，但一個服務持續在市場上與其他對手競爭，大部分的開發團隊成本是在服務上線之後的維護。
Site Reliability Engineering 主張 Developer 以及 Operator 對維持服務的可靠性有共同責任，應該要共同合作達成。透過定義服務的可靠性以及可靠性的等級來量化服務的可靠程度，以及在處理錯誤及推進新功能間取得平衡。

在分散式系統的盛行之下，我們很難用 可靠性=服務良好的時間/服務總服務時間 來計算可靠性，一個比較有代表性的算法是 與使用者正常的互動數量/與使用者的總互動數量。
關於定義可靠性一個重要的觀念是，基本上不可能定義 100% 的可靠性，愈逼近 100% 可靠所要花費的努力的比例與得到的可靠性是愈不成比例的。這會讓團隊減緩新功能的發布，進而減緩了業務的推動。這也就是為什麼大部分的服務提供商的承諾的可靠性都是 99.x...% 的原因。通常承諾的服務可靠性是以 SLA 的形式呈現，而如果服務的 SLA 是 99%，那代表團隊有 1% 的犯錯是可以被接受的，這被稱為 錯誤預算(error budget)。

SLA 是一種與客戶的合約，在提到 SLA 之前，要先提到 SLI 以及 SLO，這三者的關係如下

SLI(Service level indicator) 量化了服務對於客戶的服務級別，
SLO(Service-Level Objective) 定義了期望維持的 SLI，
SLA(Service-Level Agreement) 則是基於 SLO 制定的合約。

SLI 是從用戶的角度來看服務的可靠性，比方像是 正確的請求數/總請求數 或是 服務正常的時間/總服務時間。
SLO 則是制定一個與 SLI 對齊的目標，比方像是 每個月的 正確請求數/總請求數 比例應該 > 99%。這通常是團隊的內部指標，需要團隊內的成員達成共識。這對於團隊評估使用者的是否正常使用服務是一個重要的指標，如果服務持續在 SLO 內，代表服務目前仍然能讓大部分使用者正常使用，團隊可以繼續加速推進功能發布，抑或是預期可能的錯誤發生提前優化系統，反之則代表錯誤預算用盡了，團隊應該將重點放在可靠性的改善而非持續開發新功能。
SLA 是一種對客戶的承諾，這通常是和客戶簽約的，通常會是 每年(每月)的停機時間不得低於幾分鐘 之類的內容。

接受錯誤的發生

錯誤是無可避免的，透過建立監控以及觀測系統的機制，SRE 可以在錯誤事件發生時很好的透過數據掌握現有系統的問題。
而在錯誤事件結束後，了解錯誤事件的發生是重要的。透過在事件解決後做一個不責怪的檢討並記錄原因，可以讓團隊更好的制定下一步的改進計畫以預防事件再次發生。
這樣的 回顧活動(retrospective) 我們稱為 不責怪的事後檢討(blameless postmortem)。

一個不責怪的事後檢討只關注事件，不指責特定的人或團隊及其行動或行為。每個在事件發生時發揮作用的人都該是做事後檢討成員的一部分，如此一來團隊才能更好的收集訊息。
不責怪在事後檢討是非常重要的，營造出心理安全的事後檢討環境能幫助團隊成員相信說出想法、問題、疑慮以及錯誤不會受到逞罰或羞辱，抑或是擔心自己被指派成提出改進項目的負責人。
當團隊成員都不願意提出自己的想法或意見時，會導致團隊的學習與創新機會逐漸消失。

一個事後檢討紀錄會包含以下項目

  • 事件發生的 時間 以及 細節
  • 為了減緩或解決事件採取的 行動
  • 事件的 影響範圍
  • 事件發生的 根本原因
  • 防止再次發生事件的 後續行動

一個良好的事後檢討可以幫助團隊

  • 確保團隊了解完整個錯誤起因
    • 一個錯誤事件的發生可能是多個根本原因組合在一起造成的，妥善的執行事後檢討以及紀錄內容能有效的釐清事件根本起因以及其解法，並將知識傳遞給整個團隊。
  • 定義有效步驟預防再次發生
    • 在事件發生的當下，團隊可能做了一些處理，但在短期或長期這個事件可能會再次發生，團隊需要防止錯誤事件再次發生，並為此預防工作做出排序。
  • 減少錯誤發生的可能性
    • 每次的錯誤事件對團隊來說都是一種壓力，從錯誤中學習，從而避免錯誤的發生及再次發生，讓團隊可以花費更多的力氣在改進系統上。
  • 避免複雜性
    • 大部分發生錯誤事件時我們很常用的是短解，這樣的修復就像是衣服上的補丁一般，如果沒有良好的事後分析並做長解，隨著系統不斷的功能疊代，修復工作可能會更加困難，系統的可維護性會大大降低。
  • 從錯誤中學習
    • 每次的錯誤都是一個良好的學習機會，透過事後檢討的共享，這個錯誤為整個組織提供了額外價值。專注於不責怪的檢討讓團隊完全專注於回顧事件並防止事件再次發生。

小步疊代

Site Reliability Engineer 注重小步疊代，透過發布小的疊代可以降低發生錯誤時對使用者的影響，然而發布是有成本的，頻繁的發布意味著花費更多的時間在做發布相關的準備工作。透過 CI/CD 可以很好的降低發布版本的相關成本。 Continuous integration 通常是指在開發環境中建置、整合並測試程式碼，目的在於幫助 Developer 能更頻繁的進行程式碼的開發及測試工作。透過不斷地及整合程式碼提前發現問題，增加程式碼的品質。 Continuous delivery 指的是只要業務上的需要，團隊可以頻繁的發布版本到生產環境。
透過 CI/CD 的實現，很好減少了人為錯誤，也處進了更高的程式碼品質，在錯誤發生的時候也更容易恢復正常。

一種方法是透過金絲雀部屬，這個名字取自於以前煤礦工人會把金絲雀帶到煤礦坑中，用來探測是否有危險的氣體，如果金絲雀死亡則代表有危險。
金絲雀部屬意在透過將新的功能疊代交付到一小部分的用戶，來驗證交付是否正確，避免大規模的使用者遭遇異常。

善用工具以及實作自動化

在日常工作中，很難避免會有一些雜務(toil)，一個雜務的定義通常是

  • 需要手動的
  • 不斷重複的
  • 可被自動化的
  • 沒有持久價值的
  • 隨著使用者增長業務量會跟著增加的

絕大部分軟體開發團隊的成員不會喜歡做雜務，像是重複的登入一台伺服器取得檔案等等。
過多的雜務會影響團隊的士氣及產出，Site Reliability Engineer 會嘗試使用方法來降低雜務，當發現工作可以自動化完成時，著手進行自動化流程的開發。
由於雜務因為複雜度及時間等因素，很難完全消除，一個可行的做法是定義雜務佔工作中的比例來決定工作的優先序。
透過優化或自動化雜務來降低手動重複的工作來讓團隊能更專注於有價值的工作，進而提高團隊的生產力。

測量一切工作

測量一切工作有助於團隊了解服務的現況

通常來說測量的目標有三個

  • 測量可靠性
    • 如同上面提到的通過 SLI 以及 SLO 來測量可靠性,一個好的 SLI 要能跟使用者的體驗建立關聯，所以 SLI 的測量最好以 使用者的觀點 出發。
      比方說，多台應用服務伺服器中的其中一台伺服器處於不可用狀態使用者並不在乎，這對他們來說並不重要。而使用者試圖開啟一個網頁卻發現要等很久時，使用者就會不開心。
      所以定義出一個可以反映出使用者目前在使用服務是緩慢的指標，會比單純的 CPU 使用率以及 記憶體使用率更好。
  • 測量雜務
    • 通過測量各類型工作的比例來調整工作優先序以及優化工作效率，並持續追蹤測量結果。
      第一步我們會需要定義出什麼是雜務，以開發團隊來說，通常是開發團隊的團隊成員。再來我們要找出一個單位，像是這個雜務會花費一個團隊成員多少分鐘等。
      最後要持續測量結果，我們才能得知在透過像是工具或是自動化來消除或改進雜務有真正的得到提升。
  • 監控
    • 監控系統幫助我們更好的了解服務目前的健康狀態，透過設定警報以及建立儀表版讓團隊可以更好的提前察覺問題並處理。大部分常見的警報實作會像是 CPU 使用率以及 記憶體使用量等。
      但如上所述，我們應該關注於使用者所關注的事情，像是使用者能不能在一秒或兩秒內看到一個網頁。要關注於讓使用者不開心的 症狀 而不是 原因。
      為每一個原因建立一個單獨的警報會導致大量的 "狼來了"。意即這是沒有用的警報。
      想像一下你正在用電腦玩一個遊戲，記憶體跑到了 90%，但你的遊玩體驗或許還是順暢的。同理，這類單獨的基礎指標並不能充分反映使用者在使用服務時的體驗。
