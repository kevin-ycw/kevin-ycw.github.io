版控使用的目的

  • 在什麼時間(When)，
  • 修改了什麼內容(What)
  • 是誰修改的(Who)
  • 為什麼要修改(Why)

主流的版本管理系統

  • 集中式版本控制系統
    • 單一節點集中管理，有單點損壞風險
    • 提交操作必須有網路
    • 客戶端沒有保存所有內容，網路不佳環境容易同步失敗
  • 分散式版本控制系統
    • 提交操作都在本地進行，向其他人提交或與遠端同步才會需要網路
    • 主流是 Git

版本控制系統的基本觀念

  • 程式碼倉庫(codebase)
  • 分支(branch)
  • 主線(trunk)
  • 版本號碼(revision)
  • 標籤(tag)
  • 頭(head)
  • 合入(merge)
  • 衝突(conflict)

常見的分支開發模式

  1. 主線開發，主線發佈(trunk-based development & release)
    • 像主線提交程式碼，從主線做軟體交付
    • 可以分為
      • 高頻交付
        • 每天一次或數次
        • 需要較完善的交付基礎設施
      • 低頻交付
        • 週期較長的大型軟體開發專案
        • 主線程式碼可能長期處於不可用狀態，只有在專案內所有功能的程式碼開發完成後，才開始進行軟體調整和整合測試工作
      • 優點: 分支方式簡單，容易管理
      • 缺點: 不是團隊所有人都在做缺陷修復，如果主線正在修復缺陷且處於不可用狀態會有一定的資源浪費
      • 提交未完成功能到主線程式碼在發佈後要不影響用戶的正常使用 => 軟體發佈跟功能發佈應分開來看
  2. 主線開發，分支發佈(trunk-based development & branch-based release)
    • 流程
      • 開發人員將寫好的程式碼提交到主線
      • 開發完成或準備做版本發佈時拉出一條分支
      • 在這個新的分支上進行整合測試並修復缺陷，品質達標後對外發佈版本
    • 特點
      • 主線程式碼提交活動頻繁，對保障主線程式碼品質有較大挑戰
      • 分支只修復缺陷不增加新功能
      • 新功能發佈後發現功能缺陷一樣在該分支修復，並將修改合併回主線
      • 一種方式是在主線上修復，然後挑出修復的程式碼到缺陷所在分支
      • 品質打磨週期(Branch Stabilization Time): 分支程式碼達到可以交付狀態的時間週期
        • 可以做為評估主線程式碼品質指標
        • 品質打磨週期愈短，說明主線程式碼品質愈好
        • 品質打磨週期極短時可轉換到高頻 主線開發主線發佈模式
      • 優勢:
        • 人員可以持續在開發主線上開發，不受版本發布影響
        • 新發佈版本出現缺陷後，可以直接在自己版本發佈分支上進行修復，不會受到開發分支正在開發新功能的影響
      • 劣勢:
        • 主線上的程式碼只能針對下一個發佈版本做功能開發
        • 分支沒有好好管理的話很容易出現分支地獄
  3. 分支開發，主線發佈(branch-based development & trunk-based release)
    • 流程:
      • 團隊從主線拉出分支，並在分支上開發軟體新功能或修復缺陷
      • 分支上的功能開發完成後要對外發佈版本時，才合入主線
      • 在主線進行缺陷修復，品質達標後再將程式碼打包發佈
    • 優勢:
      • 分支合併前各分支開發活動不受影響
      • 團隊可以自行決定發佈哪個分支
      • 新版本出現缺陷時可以直接在主線上修復或使用 hotfix 分支修復
    • 劣勢:
      • 為了避免分支之間受影響，開發人員可能減少合併回主線的時間，這會造成各分支程式碼衝突，開發人員的重構意願降低，進而降低了程式碼可維護性
      • 分支過多可能因為部分分支生命週期過長，程式碼合併及驗收成本會快速增加，成本與在生命週期期間合到主線的分支數成正比
    • 好的實現:
      • 主線盡可能維持在可發佈狀態
      • 每個分支的生命週期盡可能短
      • 主線程式碼盡早與分支同步
      • 一切以主線程式碼為基準，盡可能不要再各功能分支做合併
    • 二種不同的實作
      • 特性分支模式
        • 一個特性一個分支
        • 特性開發完成後立即合入主線
        • 目的:
          • 讓團隊以 特性 這個層次上平行工作
          • 保持主現在穩定可發布狀態
        • 優勢: 每次發布內容調整容易: 新功能或者缺陷在版本發佈時間點之前無法完成時，則不必合入主線
        • 不足:
          • 特性分支過多會有較高合併分支成本
          • 主線品質驗證通過時，其他分支此時都應該從主線拉取最近通過品質驗證的新程式碼
        • 多個特性同時開發的極端作法:
          • 所有已完成特性分支合併到主線，並讓全部分支達到可發佈狀態
            • 問題: 可能較難定位跟修復發現的缺陷
          • 排順序合併到主線:
            • 問題: 多特性分支排隊合併，造成部分分支可能等待較長時間
        • 特性分支好的實踐:
          • 特性分支的生命週期盡可能短
          • 分支的開發測試工作盡可能在 3 天內完成
          • 特性盡可能拆小
          • 開發人員要每日將開發中的特性分支合併主線
      • 團隊分支模式
        • 一組人在同一個分支上進行開發工作
        • 常出現在規模較大的團隊(40 人以上)共同開發一款產品，常見於通訊公司的產品研發或大型客戶端軟體研發
        • 多個團隊開發不同的系統元件，只有當一系列功能特性開發完成後，才對外發佈新的軟體版本
        • 很容易變成瀑布式開發
        • 成功應用此模式的關鍵:
          • 每個團隊盡早向主線合入高品質程式碼，即便沒有要馬上發佈
          • 向主線合入程式碼後，盡快使其達到可交付狀態
          • 其他團隊盡早從主線拉取可交付狀態程式碼，與自己的分支合併

分支模式的演化

三駕馬車分支模式

開發團隊僅維護三個分支

  1. 開發分支
    • 所有開發人員提交程式碼的目標分支
  2. 預發佈分支
    • 從開發分支得到預備發佈的功能
    • 在預分發佈分支上進行缺陷修復、檔案產生等發佈相關工作
    • 不做新功能開發
    • 達到 Alpha 版本發佈品質時會發佈一個 Alpha 版本給極少數用戶進行體驗
    • Alpha 版本穩定後發佈 Beta 版本提供嘗鮮用戶進行版本體驗，以便盡早發現可能存在的品質問題
  3. 發佈分支
    • Beta 版本穩定後會發佈 RC (Release candidate)版本
    • RC 版本穩定後即可作為正式版本發佈

Gitflow 分支模式

  1. Master 版本是正式版本的發佈分支
  2. Release 分支是用於品質打磨的預發佈分支。Release 分支品質達標則可合併到 Master 分支，並且同步合併到 Development 分支
  3. Development 分支是對新功能進行整合的分支
  4. Feature 分支是為了開發某一功能特性，開發人員從 Development 分支上拉出的分支
  5. 如果已發佈分支出現缺陷，則從 Master 分支的上一個發佈版本切出 Hotfix 分支，修復完成後再次合併回 Master 並發佈新的版本

特色:

  • 特性分支模式和三駕馬車模式的組合
    • 優點: 每個分支定義明確
    • 問題: 分支多(同特性分支模式問題)

GitHub flow 分支模式

  • 源自於 GitHub 團隊的工作實踐
  • 對開發者的開發紀律要求比較嚴格
  • 對品質保障手段的要求也比較高
  • 開發新特性或修改缺陷的工作步驟
    1. 從 Master 分支上建立一個新的分支，以這個特性或缺陷的編號命名
    2. 在新建立的分支上提交程式碼
    3. 功能開發完成並自己測試通過後發 PR
    4. 其他開發人員對 PR 進行審查，確認品質合格後合入 Master
  • 如果特性分支存在的時間很短，則該模式可被認為是高頻的 主線開發主線發佈 模式

分支策略的選擇

版本發佈模式

三種基本的版本發佈模式

  • 版本發佈模式三要素: 相同的 3 個約束變數
    • 交付時間點(schedule)
    • 特性數量(features)
    • 交付品質(quality)

在團隊資源相對固定的情形下，只能對其中二個因素提出固定的要求

  1. 專案制發佈模式(Project Release Mode)
    • 定義
      1. 預先確定某一版本需要達到的功能特性
      2. 只有當該集合內的所有特性全部開發完成並達到相應發佈品質標準後，才能發佈該版本
      3. 前後兩次發佈之間的時間間隔沒有明確規定
    • 這是一種古老的發佈方式
      • 目標:
        • 針對一個特定版本確定版本中的特性數量和品質標準，再估計版本交付週期
        • 固定特性數量和品質要求，藉此相對固定了交付時間
    • 好處: 確切知道每個版本包含哪些具體功能，有助於商業套裝軟體售賣
    • 不足: 專案整個交付週期較長=> 專案制發佈模式須將所有需求全部實作完成後才能一起發佈
  2. 發佈火車模式(Release Train Mode)
    • 常見於大型套裝分發類軟體
      • 大型傳統軟體企業通常有多產品線且有複雜的相互依賴
      • 解決協同發佈議題: 制定每個版本的發佈週期，所有參與到該版本開發的團隊必須對齊該版本的各開發階段 => 每個版本都像一列火車，事先計畫好甚麼時間點發車
      • 當版本發佈週期決定後，發佈管理團隊通常與負責個產品開發的團隊進行提前溝通，討論要發佈哪些內容，並將結論發佈在企業版本表中
      • 提前制定時間表的目的是讓各種業務和技術部門有足夠多的時間進行預計畫，以便做出依賴和影響相關的評估工作
      • 制定發佈計畫是一種正式的結構化過程，包括
        • 發佈詳細資訊，如
          • 相對標識
          • 名稱
          • 部署日期
          • 風險等級
          • 發佈類型
          • 企業
          • 計畫
          • 投資組合
        • 每個階段要完成的活動和任務
        • 里程碑時間和品質要求
        • 負責管理發佈火車的主要負責人
      • 好處: 透過平行多火車的模式，將突發需求排入某一列發佈火車，用戶可提前體驗最新產品版本所提供的新特性
      • 可能問題: 參與團隊的人數較多時，溝通協調成本就會較高
  3. 城際快線模式(IntercityExpress Mode)
    • 固定時間和品質的維度，且時間週期相對較短(如一週，甚至一天或更少)
    • 針對在發佈時間點已達到品質標準的特性進行一次發佈
    • 與發佈火車的差別
      1. 發佈週期間隔較短，通常在二週以內
      2. 負責特性開發的團隊可以自己選擇搭乘哪列城際快線，而不必提前很長時間確定下來
    • 常見於提供網際網路服務或 SaaS 服務的軟體公司
      • 好處:
        • 減少了團隊及角色之間的協調成本，因為每個人都事先知道每次發佈的具體時間點
        • 即使某特性沒有管上最近一次的版本發佈，團隊也確切知道這個特性是否可以在下一次發佈時間點對外發佈
        • 每個人都非常清楚各特性發佈時間點
        • 團隊更聚焦於生產品質
      • 不足之處:
        • 發佈頻率較高，因此未完成功能也會一同發佈出去
        • 對程式碼提交品質要求較高，需要強大的品質基礎設施保證
      • 發佈週期的建議: 在不影響用戶體驗、不增加成本且合乎規範的前提下，盡可能縮短到適度緊湊

分支策略與發佈周期的關係

  • 軟體開發週期極長的 專案制 + 軟體發佈頻率極高的 城際快線式 的團隊會使用 主線開發主線發佈 的分支策略

    • 分支策略沒有絕對，會受到團隊成員人數、產品架構和品質保障基礎設施狀況的影響
    • 專案制發佈模式不會消失，每個新產品的在完成第一個可推廣版本前，可能都需要一個首次啟動程序

  • 城際快線模式是 持續交付 2.0 所提倡的模式

    • 在版本發佈周期較長的企業也可以在專案至發佈模式中套用城際快線模式
      • 在專案週期內加入固定時間的疊代，並要求在每個疊代結束都能得到可交付狀態的產品
        • 可交付狀態指的是已完成的軟體特性達到可發佈的品質標準，並非商業化發佈

  • 發佈週期短的時候，主線開發模式會更具有優勢，因為減少了分支開發模式的分支合併成本，如果發佈週期短於二週，則主線開發模式會是較好的方案

小結

每種分支策略都有其優勢與劣勢，目前在趨勢上，軟體發佈頻率愈來愈高且發佈週期愈來愈短

持續交付 2.0 提倡鼓勵持續整合的分支策略，選擇分支模式的原則有以下幾條。

  1. 分支愈少愈好，最好只有一條主線
  2. 分支生存週期愈短愈好，最好 3 天以內
  3. 在業務允許的前提下，發佈週期愈短愈好
