Amazon API Mandate

  1. 從今天起，所有的團隊都要以服務介面連接埠的方式，提供資料和各種功能。
  2. 團隊之間必須通過連接埠來通信。
  3. 不允許任何其他形式的互操作：不允許直接連結，不允許直接讀取其他團隊的資料，不允許共享儲存空間，不允許任何形式的後門。唯一許可的通訊方式，就是透過網路呼叫服務。
  4. 具體的實際技術不做規定，HTTP、Corba、Pub/Sub、自定義協議皆可。
  5. 所有的服務連接埠，必須從一開始就以可以公開作為設計導向，沒有例外。這就是說，在設計連接埠的時候，就預設這個連接埠可以對外部人員開放，沒有討價還價的餘地。
  6. 不遵守上面規定者，一律開除。

將單體系統變成服務導向架構後(Monolith -> Service-Oriented Architecture)，
AWS 可以在一小時內部署 1079 次，
在團隊合作上達成了 增加小團隊內部溝通，減少團隊間的溝通成本。

大系統小做

持續交付架構要求

  1. 為測試而設計(design for test): 在寫程式考慮程式的可測試性
  2. 為部署而設計(design for deployment): 在開發功能時要思考怎麼部署新功能
  3. 為監控而設計(design for monitor): 在功能開發的設計階段要思考上線後怎麼監控
  4. 為擴展而設計(design for failuree)
    1. 支援團隊成員規模的擴展
    2. 支援系統自身的擴展
  5. 為失效而設計(design for test): 部署或發布失敗時的退版方案

系統拆分原則

  1. 每個元件或服務要有清晰的業務職責，可以被獨立修改甚至被另一種實作方案所替代
  2. 高內聚低耦合: 每個服務知道盡可能少的資訊，才能達到容易維護
  3. 系統的建置跟測試要簡易，才能達到持續交付
  4. 要使團隊成員之間的溝通協作更加順暢

服務拆分看起來很美好，但由於系統是由多個服務組成，會導致搜尋問題的難度增加。
因此在服務拆分的同時，也要建立相應的建置、測試與部署和監控機制

常見的架構模式

微核架構(microcore architecture,plugin architecture)

主要業務功能和業務邏輯都透過外掛實作，適合客戶端軟體的開發

優點:

  1. 良好的功能延伸性: 新功能都是開發新外掛即可
  2. 容易發佈: 外掛可以獨立加載卸載
  3. 容易測試: 功能之間是獨立的，可以對外掛進行隔離測試
  4. 可自訂性
  5. 漸進式開發: 按需開發

缺點:

  1. 擴展性差: 核心通常是獨立單元，不容易做成分散式
  2. 開發難度相對高，需考慮外掛跟核心的通訊機制及外掛登記機制
  3. 高度依賴框架

微服務架構(microservice acthitecture)

  1. 單一應用程式劃分成一組小的服務
  2. 每個服務執行在獨立的處理序中，用輕量級的通訊機制通訊
  3. 每個服務都圍繞著具體業務進行建置，能獨立部署到生產環境及測試環境等

優點:

  1. 擴展性好: 服務之間低耦合，可按需單獨擴大部分服務
  2. 容易部署: 每個服務是獨立可部署單元
  3. 容易開發: 每個服務可單獨開發及單獨部署
  4. 容易單獨測試: 修改指涉及一個服務，就只要測試一個服務

缺點:

  1. 系統依賴大量微服務導致凌亂及笨重，網路通訊消耗也會變比較大
  2. 一個外部請求涉及很多內部服務通訊，問題除錯診斷困難
  3. 原子操作不好處理
  4. 跨服務組合業務測試困難
  5. 公用類別庫升級管理困難

單體應用(monolithic application)

優點:

  1. 開發除錯簡單
  2. 部署簡單
  3. 擴展容易
  4. 對開發人員的系統設計及開發技能要求較低

缺點:

  1. 容易產生混亂程式碼
  2. 難與新技術共同使用
  3. 只能擴展整個應用
  4. 持續部署困相對較難

架構改造實施模式

架構改造需求的 3 種實施方法

拆遷者模式

根據目前業務需求開發一個全新的版本，一次性完全替代原有的遺留系統

好處:

  1. 沒有歷史包袱，可以按預期進行架構設計

風險:

  1. 業務需求遺漏
  2. 開發速度較慢，可能無法因應市場環境變化
  3. 必須分出人力同時進行舊版本維護及新版本開發，人力資源消耗大
  4. 新版本上線後可能無法滿足業務需求

絞殺者模式

保持原有遺留系統不變，當需要開發新功能時，重新開發一個服務，實作新的功能。透過不斷建置新服務逐步取代遺留系統，最後就可以替代遺留系統。

好處:

  1. 不會遺漏原有需求
  2. 可以穩定提供價值，頻繁交付版本可以持續監測成果
  3. 避免 閉門造車

壞處:

  1. 架構改造時間跨度大
  2. 產生一定的疊代成本

修繕者模式

遺留系統的部分功能與其他部分隔離，以新架構進行單獨改善

好處:

  1. 系統外部無感受
  2. 不會遺漏原有需求
  3. 隨時可停下改造工作，響應高優先級的業務需求
  4. 避免 閉門造車

壞處:

  1. 架構改造時間跨度會變大
  2. 額外的架構改造疊代成本

資料庫拆分

大部分來說，單體應用的最大耦合點是資料庫。
資料拆分可以遵循以下步驟:

  1. 詳細了解資料庫結構
  2. 先拆分資料庫
  3. 資料雙寫無誤後找到出服務的縫隙
  4. 將拆分的流程模組和資料庫組合成微服務

小節

3 種常見的軟體架構模式

  1. 微核架構適合於客戶端軟體
  2. 微服務架構適合於大型後台服務端系統
  3. 單體應用適合創業公司或中小型專案

對遺留系統進行架構改造的 3 種方式:

  1. 拆遷者模式: 一次性改寫所有程式碼
  2. 絞殺者模式: 在幾乎不改遺留系統的情形下，透過增加新的服務來不斷替道遺留系統的功能
  3. 修繕者模式: 對原有系統逐步改造
