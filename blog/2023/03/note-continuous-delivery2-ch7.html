部署流水線是持續交付 1.0 核心模式，展現了從

  • 程式碼提交
  • 建置
  • 部署
  • 測試
  • 發佈

的整個流程

GoCD 的部署流水線

  1. 提交建置-自動觸發
    1. 編譯打包
    2. 程式碼規範靜態掃描
    3. 執行單元測試及整合測試
  2. 次級建置-自動觸發
    1. 端到端測試
  3. UAT 部署
  4. UAT 結果
    • 手動測試驗證完成後標記為 驗收通過
  5. 性能測試
    • 自動化性能測試
  6. 內部體驗
    • Alpha 版本部署到內部伺服器給其他團隊試用
  7. 外部體驗
    • Beta 版本發給外部企業用戶體驗
  8. 上傳發佈
    • 將確定的商業發佈版本上傳到指定伺服器供用戶下載

部署流水線的設計與使用

  1. 流水線的設計原則: artifact 重複被部署
  2. 與業務邏輯松耦合: 部署流水線應該與具體部署建置業務松耦合，像是部署或建置的腳本應該放在該產品的程式碼倉庫中
  3. 平行化原則: e.g:在建置時同時跑自動化測試
  4. 快速回饋優先: 比較快得到回饋的任務先執行，像是單元測試優先於整合測試及效能測試
  5. 重要回饋優先: 更重要的回饋任務要考慮更優先，像是一個單機的軟體，其安裝測試就會比單元測試有價值，畢竟不能裝的軟體沒有價值

團隊協作紀律

  1. 立即暫停原則: 部署的某個環節出問題導致執行失敗時，團隊應該立即停下手中任務並安排人員著手修復。在問題修復前都不該有修復問題外的提交程式碼。
  2. 安全審查原則: 角色協作時，程式碼或軟體套件應該來自有被審查的環境，部署流水線的任何環節都應該只使用來自部署流水線的 artifact。任何的 artifact 都應該接受受控管理，像是:
    • 測試人員不應該私自拉取程式碼並手動建置軟體套件進行測試後建置軟體套件進行測試
    • 不接受開發人員透過非正常發佈方式傳遞軟體套件進行測試。
    • 驗證的交付物須確保來自可信來源，例如統一版控或 artifact 倉庫。

部署流水線平台的構成

  1. 唯一受信源
    部署流水線的原材料及製品保存。
  2. 部署流水線平台
    負責各種任務調度與結果統一展現，會與各式工具或系統互相協作完成整個交付流程。
  3. 基礎支撐服務層
    由多種工具組成，提供軟體的建置、測試和部署等基礎能力。

部署流水線可以

  1. 準確展示部署流水線個環節狀態
  2. 自動收集各還決產生的衡量資訊，並對價值流動效率進行度量
  3. 事件追溯能力: e.g: 什麼人在什麼時間執行了什麼操作?
  4. 對部署流水線產出物的追溯能力，像是
    1. 部署流水線的任意產出務
    2. 對應來源
    3. 建置腳本與環境
  5. 重新產出 artifact 的能力: 可以從歷史紀錄產出某個版本的 artifact

編譯建置管理服務

建置管理服務包含

  1. 建置任務管理
  2. 建置任務調度
  3. 建置群集管理
  4. 建置執行器

任務管理服務會包含

  1. 接收子服務: 接收建置任務請求並記錄請求相關資訊，並將其加入待建置佇列中。
  2. 通知子服務: 通知建置任務發起者任務的相關結果

調度服務負責根據調度算法選擇建置任務，並將其發送到相應的建置機器上執行建置。

群集管理服務負責對各類建置環境的管理，像是建立不同的建置環境與銷毀以及環境的狀態管理。

執行器是執行建置任務的代理，一個群集會有很多執行器，根據任務資訊從對應的程式碼倉庫取得程式碼，並透過護轎對應的建置工具來完成建置任務。

自動化測試管理服務

  • 測試任務管理服務: 用於接收任務和回饋任務的執行結果。包含
    • 任務接收子服務
    • 任務回饋子服務
  • 測試用例調度服務: 負責根據一定的調度算法從任務管理服務中選擇測試任務執行。分成
    • 順序執行測試用例
    • 併行執行測試用例
  • 測試集群管理服務:
    • 測試環境的建立與銷毀
    • 測試環境的狀態管理
  • 用例健康管理自動化服務: 如果有大量自動化測試用例的公司會有需要監測用例的健康狀況。例如按照依訂規則來自動判斷，如果被判斷為不穩定用例則移出用例集

軟體佈署管理服務

軟體在開發到佈署時會碰到的幾個障礙:

  1. 測試環境跟正式環境用的是不同的技術堆疊: 例如 J2EE 伺服器是要額外付費的，而 Tomcat 是不用付費的。語法檢查上 Tomcat 也較為寬鬆。有些企業為了省錢會在測試環境用 tomcat，在正式環境才用付費伺服器。這樣就會導致錯誤在上線後才發現。
  2. 產品研發部門及維運部門的責任衝突: 維運部門負責正式環境穩定性; 產品研發團隊負責開發新功能。

在持續交付的工作模式下，所有人都該使用相同的工具集。任何人在被授權後都該可以用同樣的方法佈署環境。

基礎環境管理服務

  • 建構管理
  • 測試管理
  • 佈署管理

企業製品庫管理

製品庫(artifact repository)分為

製品庫企業內部企業外部
臨時正式
軟體包庫ABX
鏡像庫CDY
  • 臨時軟體包庫 A:
    1. 每次觸發建構後產生的二進位包
    2. 不能直接被佈署到生產環境
    3. 容量不足時可清除
  • 正式軟體包庫 B:
    1. 經過佈署流水線驗證能發布到正式環境的軟體包
    2. 從臨時軟體包庫移動到正式軟體包庫
  • 外部軟體包庫 X:
    1. 非企業自身團隊管理和維護的軟體包
    2. 通常從互聯網或第三方機構獲得併保存
    3. 可能以二進位包保存或原始碼方式保存，也可能是僅保存外部下載連結
    4. 通常來說保存外部軟體包是為了
      1. 內部人員方便於內往取得
      2. 統一審核，內部人員僅能從被管理的庫取得
      3. 保證安全，避免用到了有安全隱患的版本
  • 臨時鏡像庫 C & 正式鏡像庫 D & 外部鏡像庫 Y: 同上，只是存放的是鏡像檔

製品庫管理原則

每一個製品都應該有唯一標誌，且可以追溯到源頭，在必要時可以透過原有的佈署流水線再次生成製品。

小結

四大基礎支撐服務

  1. 編譯建構服務
  2. 自動化測試服務
  3. 佈署管理服務
  4. 基礎環境服務

三大受信源:

  1. 需求管理倉庫
  2. 原始碼倉庫
  3. 製品庫

誰建構，誰維運: 開發人員應該要與自己開發的軟體打交道。且要在日常過程中經常與客戶聯繫，客戶的回饋還對改進服務品質至關重要。
為了達成這個工作指導原則並讓組織成為自驅動、自服務的業務導向型全功能團隊的學習型組織，會需要整個工作流程中的基礎建設支援:

  1. 基礎的研發流程自助平台，如各類運行環境的自助平台
  2. 數據自助平台
  3. 實驗測量平台
  4. 針對移動設備，建立用戶觸達平台

研發團隊要儘可能遵守以下原則:

  1. 任何軟體包的取得都須通過受控源，不該從私有渠道(Email、IM)等獲得
  2. 盡可能將一切流程自動化，並持續優化執行時間
  3. 每次提交都能夠自動觸發佈署流水線
  4. 盡可能少用手動觸發方式
  5. 必須執行立即暫停原則
