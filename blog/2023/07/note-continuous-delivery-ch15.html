讀書筆記 持續交付 2.0 第十五章

Authors
  • Name
    kevin wang

案例

  • 團隊情形

    • 小型團隊(不足 10 人)
    • 跨職能團隊
    • 現況為 死亡行軍: 超過預期一倍以上的軟體專案，描述按照進度表幾乎不可能完成的專案，整個專案瀰漫的失敗的氣息
    • 透過 持續改善 達成 零缺陷交付

  • 產品特色

    • 行動裝置中的系統軟體
    • 修復成本高(軟體部署在客戶端)
    • 測試左移的可行性較高(測試右移可行性低)

  • 公司情形

    • 子公司研發團隊: 500 人
    • 業務線經理(Line manager)負責多個研發組
      • 研發組有開發人員跟測試人員
    • 規劃線的產品管理團隊(Product management Team,PMT)
      • 產品經理在產品管理團隊中

  • 產品研發方法

    • 新產品研發時以專案形式將各服務線人員組成專案小組，並按照業務領域畫分子團隊
      • 技術組長為子團隊的 Scrum Master
      • PMT 團隊指派一名產品經理成為子團隊的 Product Owner
      • 子團隊之間以 Scrum of Scrum 合作

  • 改進前

    • 團隊經歷一次死亡行軍
    • 團隊情形
      • 團隊成員的研發經驗比較少且在職時間少
      • 團隊士氣低落，剛交付的專案開發一個月卻花了一個半月來修復缺陷
    • 專案研發週期: 超過六個月
    • 參與人員: 約 200 人
    • 品質要求高: 手機售賣後若有問題，召回修復成本高
    • 大型遺留系統: 10 年歷史且整個程式碼庫超過 1500 萬行

  • 改進後的無缺陷交付

    • 時間與範圍: 即便研發後期新增了最佳化需求，仍舊按時間完成研發任務
    • 產品品質: 負責開發的所有功能在交付後，經驗證未發現任何缺陷
    • 協作流程: 團隊建立的新的軟體研發工作實踐
      • 需求分析(使用者故事) 與 相對估算方法
      • 建立以小需求(使用者故事)流動為主的看板實踐，不再追蹤個體活動與任務
      • 使用主線開發模式
      • 開發人員編寫自動化單元測試
      • 每個小需求拆分多個開發任務進行
      • 每個任務完成即向主線提交程式碼
      • 每個小需求完成後即進行需求功能驗收測試

改進方法論

大團隊與小團隊的改善路徑會不同

  • 大團隊從整理結構入手更加有效
  • 小團隊應該更關注團隊內部的持續改善

指導思想

  • 目標驅動，從簡單問題開始，持續改善
  • 目標
    • 時間與範圍: 在規劃時間表內，完成該業務領域的功能特性開發
    • 品質: 產品研發品質一次性達標
    • 流程: 建立良好的工作流程習慣，團隊經驗可以固化

試點團隊的選擇

通常考慮以下四點

  1. 專案壓力適中，有相對富裕的時間(slack time)
    1. 專案時間壓力大時團隊成員通常會拒絕一切改變
    2. 沒有交付壓力則看不出試與不試的差別
  2. 團隊負責人心態開放，能承受壓力勇於嘗試
    1. 改變對於團隊的每個人都是挑戰，但對團隊的負責人挑戰最大
  3. 團隊成員有熱情，對即將進行的改變願意積極思考且給予互動與回饋，並在試點結束後有強烈的自豪感及成就感
  4. 業務方有頻繁的快速交付訴求，如果業務方在這方面的訴求不高，即便團隊掌握了各項技能，由於得不到回饋，快速交付的成效就不高了

第一階段: 研發準備期

Product backlog 的特點:

  1. 由 Epic 以及 User Story 組成
  2. 每個 User Story 都是由 As a user, I'd like to do xxx.
  3. 每個使用者故事都有一個業務優先級

功能簡介與需求拆分

  1. 團隊召開產品功能介紹會，由產品經理講述功能需求，並將功能列表進行拆分與重組
    1. 需求範圍太大不利於估算的準確以及各職能(產品人員、技術人員及測試人員對於需求的理解與溝通)
    2. 估算不會在功能簡介及需求拆分階段執行
    3. 團隊評估認為需要再釐清並研究使用者故事時會做 Spike Story。

架構設計與需求依賴識別

在案例中，需求拆分的同時團隊進行了一些設計上的討論，透過門面模式(Facade pattern)來將遺留程式碼及新功能實作隔離。

工作量估算與排期

用點數估算取代時間單位的相對大小估算，這種估算法根據了以下的假設

  1. 開發活動永遠應該是整個流程中的瓶頸，如果需求完成後可以很快完成測試工作，那麼相對開發時間來說，測試獨佔週期可以忽略不計。
  2. 從功能需求開發的整體工作量角度來看，大部分來說需求開發 & 測試 & 審核的工作量是差不多的。所以工作量估算時可僅考慮需求開發的工作量。

估算會有偏差，需求的大小合適且工作量小於 1 人週時，對於較長週期的軟體專案來說偏差是可接受的。

經過需求拆解與估算後會得到使用者故事列表以及工作量，在案例中有 39 個使用者故事以及 120 點，接下來是預估計算專案完成的時間，需考慮

  1. 最初 3 個疊代工作的產出可能會有所下降，因為需要做一些準備工作，如持續整合伺服器或自動化建置腳本等。
  2. 由於前置的準備，在 3 個疊代後工作速度會提升。
  3. 雖然有自動化測試等保護會較有信心，但技術組長仍舊加了 1.5 個疊代的緩衝時間。
  4. 以不加班作為估時的準則，並與團隊成員協調期間內盡可能不做休假安排

第二階段: 軟體交付期

需求拆分、工作量估算以及專案排期完成後，接下來就進入了正式的軟體交付期。透過引導團隊識別問題、持續改善，讓團隊成員共同參與流程改進與建設，進而實作團隊自組織。

透過可視化看板改進工作流程

看板方法是一種發現流程問題的有效方法，透過可視化目前的流程狀態識別存在的問題，並快速制定和實施相應的對策，進而持續改進，達到提升效率的目標。
以下是流程可視化能識別以及解決問題的過程。

  1. 識別壞味道

    1. 每個疊代可以交付的需求數可能因為需求粒度大有以下壞味道
      1. 團隊只能做 1 ~ 2 張工作量很大的使用者故事
      2. 開發人員和測試人員會以瀑布式開發的方式做任務分解

    壞味道會引起 3 個問題

    1. 整合時間太晚: 只能在疊代末期整合開發人員的程式碼。整合時間太後面，品質問題不可控。
    2. 測試人員的測試案例在疊代後期整合時間才能用到。

  2. 讓價值流動起來

    • 透過看板來看任務的流動速度
    • 欄目是任務的狀態
    • 拆分後的使用者故事作為工作項目放在對應的狀態欄中
    • 發現有 bug 則移動回開發欄
    • 因為外部原因導致暫時無法繼續進行的需求放在 阻塞 欄，Scrum Master 需特別關注並促進解決。
    待開發設計開發測試完成
    需求 A需求 B需求 C(Bug)需求 E
    [阻塞]
    需求 D

  3. 顯示宣告 完成 的標準定義
    案例中團隊成員都是新手，在專案啟動會議上做了兩項約束

    1. 需求啟動開發之前要寫模組設計檔案並經過審核，才能開始編寫程式碼。
    2. 編寫程式碼之後要做程式碼審核

    由於團隊成員並沒有在開發前提交模組設計檔案並經過審核，在 retro 中重新強調的這個要求，並定義了每個欄目的完成的定義

    • 待開發 -> 設計: 驗收條件或測試案例必須寫完，並經過產品人員開發人員及測試人員的共同審核，沒有異議。
    • 設計 -> 開發: 完成設計檔案的更新，設計審核完成
    • 開發 -> 測試: 編寫對應的自動化單元測試，確保所有單元測試案例成功通過，完成程式碼審核
    • 測到 -> 完成: 全部測試案例通過，所有缺陷被修復

  4. 塗鴉設計，消除浪費

    • 透過記錄每個需求在狀態的停留時間來察覺問題
      在案例中，技術組長發現簡單需求以及複雜需求在設計狀態的時間長度沒有顯著差異，在觀察過後發現新需求的設計階段是這樣執行的:

      1. 在 word 編寫模組設計
      2. 再發郵件給其他開發人員審核
      3. 等對方給她回覆郵件
      4. 根據郵件修改 word
      5. 修改完成才是完成設計

      在這樣的流程中，團隊成員會花時間在美化 word 檔，這是一種不必要的浪費，等待回應也是一種浪費 改進如下:

      1. 開發人員在白紙上畫初步設計
      2. 邀請同事到白板前畫並說明
      3. 直接提出回饋和改進建議
      4. 根據結果編寫設計檔案
      5. 提交檔案完成設計

  5. 明確狀態，關注需求的快速流動
    關注堆積問題

    • 任務在開發欄很容易有卡片堆積現象，經過了解是因為開發人員手上的需求開發完成後就會發 PR 並做下一個需求。 => 增加一個程式碼審核的狀態欄，將任務的狀態盡可能透明，讓在製品 (work in progress，WIP)在整個價值流中快速流動。 => 發出 PR 後須告知審核者，如果兩小時內沒有收到審核回覆的話需再次提醒

  6. 避免不必要的任務切換
    在案例中測試人員提出了二個不好的現象:

    1. 需求有明顯缺陷也進到了測試環境中
    2. 缺陷被提出後不論難易度都要等上一段時間才能修復

    這背後的原因是開發人員習慣忙完手中的新工作才來修復缺陷，這裡存在兩個壞味道

    1. 開發人員對需求的自測不充分，依靠測試人員找到問題
    2. 開發人員不能馬上著手修復缺陷

    根據提出問題的調整:

    • 定義新的任務狀態為 開發自測，並定義從 開發 到 開發自測 的 DoD 為重新閱讀需求卡片確認需求無誤。開發自測 到 測試 的 DoD 增加一項目為 逐條查看該需求的驗收條件

    由於在上次的程式碼審核制度調整，兩小時就能完成審核，所以任務狀態欄移除了程式碼審核的狀態

  7. 沒有回饋，就是風險
    如果有較大型需求無法拆分(拆分無法驗收或驗收成本高)，則可以將大需求拆分成多個開發任務，透過程式碼審核來確認其完成

  8. 限制在製品數量
    案例中後來發生了測試狀態的堆積問題，團隊只有一名測試人員，來不及驗證這些被修復的缺陷 應對方法:

    • 強調開發人員自測活動
    • 增加測試案例完備程度
    • 採用 精實管理理論 中的 限制在製品數量 策略
      • WIP 指的是尚未交付的半成品，還沒有產生價值

總結案例中的團隊用到的精實理論和品質管理技巧

  1. 價值流映射 Value stream mapping(故事牆是 以價值為核心的工作流程 的展現)
  2. 減少單位工作的批量大小: 將大需求拆分成更小的需求
  3. 持續改進: 不斷觀察工作流程，並發現問題，改進工作方式
  4. 限制在製品數量: 測試人員每日評估產能
  5. 品質內建: 每個環節定義了該環節的完成標準，避免不必要的任務切換

無缺陷交付

得益於不斷的驗證交付，緩衝的迭代週期最後沒有使用，團隊在驗收階段還多完成了負責人的最佳化功能需求

主線開發和持續整合

主線開發 + 六步提交法 + 架構隔離 + 自動化單元測試 來持續得到品質回饋

測試活動左移

  • 每個需求開始編寫前就要有 產品人員 & 開發人員 & 測試人員 對焦過的測試案例
    • 測試案例先行，第一時間定義軟體需求品質
  • 開發人員編寫自動化單元測試，確保程式碼品質
  • 測試人員在美個使用者故事開發完成後就立即進行驗證

程式碼審核

  • 在合併回主線時跑自動化測試任務: 六步提交法的第 4 步，第二次個人建置驗證
  • 內容檢查必須通過
  • 手動測試必須通過

關注結果更關注過程

Sprint success 很重要，而專案執行過程中的改進也很重要

小結

透過 業務需求的協作管理 來推進 分支與配置管理 及 建置與測試管理 的改進

整體改善

  • 任務拆分 => 故事拆分
  • 人天估算 => 排序法估算
  • 驟下決定的進化 => 依據顯示 PRIAD 的發佈計畫
  • 跟蹤個人任務牆 => 關注需求流動的故事牆
  • 強調 Sprint 的成功 => 弱化 Sprint 的成功的概念
  • 分支開發 => 主線開發
  • 依據故事的程式碼提交 => 依據開發任務，每人每日提交
  • 修補式開發 => 透過設計，分離關注點
  • 形同虛設的編寫規範 => 團隊認可並嚴格遵循的編寫規範
  • 最後整合的測試 => 即時測試
  • Synergy => Git
  • 增加了單元層次整合測試
  • 增加了單元測試
