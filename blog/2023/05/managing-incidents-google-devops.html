google cloud devops 筆記 - 事故管理

Authors
  • Name
    kevin wang

前言

在日常維運線上服務的時候，為了更快的發現問題，工程團隊通常會設定告警，讓團隊可以在事故可能即將發生前做預先處理，本文會對告警及事故做名詞定義，並對事故管理做展開

告警及事故的定義

  • 告警: 有不好的預兆發生，這樣的預兆可能會發生不好的事情
    • 我目前有點疲憊，精神不濟且一直打哈欠，可能快感冒了...(但也可能只是昨天沒睡好)
    • 疲勞是身體反映目前體力不佳，但這可能不是問題
  • 事故: 發生了可實際講出目前發生了不好的事情且 在什麼時間點 因為 什麼情形 而察覺到
    • 我現在發燒到 38 度，全身無力無法下床，我的發燒以及全身無力是在今天早上 7:00 醒來的時候發現的...
    • 全身無力無法下床導致無法日常生活

用戶信任建立在事故處理反應時間

用戶不會期待服務是永不故障的

在建立用戶的信任上，我們可以

  • 減少停機時間
  • 迅速回應和學習
  • 提供事件處理的透明度

事故的生命週期

一個事故的發生到被解決會經歷以下的情形

  • [事故狀態]Alerts Detected(發現告警): 代表一些不好的預兆發生了
  • Triage(分診): 判斷目前是否有事故發生?，或是否需要做預防性處理措施?
  • [事故狀態]Triaged(分診完成): 此為下一個事故的狀態，判斷為事故發生，且定義了事故的等級
  • Incident Declared(事故被定義完成): 定義事故問題及解決事故的定義
  • Working to Mitigate the Incident(緩解事故): 朝著最快緩解事故的方向努力
  • [事故狀態]Incident Mitigated(事故已緩解): 事故中被定義的問題已緩解，最好的是用戶已不受影響
  • Identifying Root Cause(分析根本原因):
    • 為什麼發生
    • 如何不再發生
  • [事故狀態]Resolved(解決): 當我們可以回答上述問題時則事故就到了事故被解決的階段
  • Postmortem Documentation Submitted(事故報告已提交): 將事故報告提交且歸檔

備註:
分診，又稱檢傷分類，是根據病人受傷情形決定治療和處理優先級的一套程序

事故處理的作法

事故發先前的最佳做法

  • 建立明確的事故宣布標準
    • 明確的標準
  • 事先選擇一個溝通管道
    • 所有相關團隊如開發團隊維運團隊及利害關係人們要知道如何做相關通訊
  • 建立溝通模板
    • 方便填寫，避免資訊遺漏
  • 培訓團隊如何應對
  • 進行演習和實踐

事故發生時要做什麼

事故處理不單只有解決具體的事故發生原因，一個良好的事故處理要包含

  • 解決事故中可被明確定義的具體問題
  • 在事故發生時持續回應事故處理的狀態

事故緩解後要做什麼

  • 不責怪的回顧(retro)
  • 不責怪的事後檢討報告(blameless postmortem)
  • follow up action item

事故反應的基本原則

  1. 保持清晰的指揮系統
    • 避免要向不同的角色(用戶或利害關係人等)回報事故進度
  2. 指定明確的角色
    • 事故回報團隊成員必須明確被定義在指揮系統中
  3. 記錄除錯及緩解的工作步驟
    1. 開始時間：故障事件發生的時間點。
    2. 觀察：記錄您對系統異常行為的觀察，例如錯誤訊息、系統日誌、性能下降等。
    3. 分析：紀錄您進行的除錯步驟，例如檢查日誌、執行命令、進行數據分析等。
    4. 緩解措施：記錄您所採取的緩解措施，例如重新啟動服務、修復故障設備、調整配置等。
    5. 結果：描述每個步驟後的結果，無論是成功還是失敗。
  4. 當事故發生時盡早且經常的宣布
    • 假如一個雲端服務發生了用戶無法登入的問題，則
      1. 事件標題：登入問題事件宣布
      2. 問題描述：用戶報告無法正常登入，遇到錯誤訊息或者連線失敗。
      3. 時間：宣布事件發生的確切時間。
      4. 受影響範圍：確定哪些用戶、區域或功能受到影響。
      5. 嚴重程度：評估事件的嚴重程度，例如高、中、低。
      6. 已經採取的步驟：描述已經進行的任何初步調查或緩解措施。

事故中的角色

  • Incident Commander (IC): 事故指揮官
    • 定義有事件發生者即是事故指揮官
    • 所有處理事故的人都必須回報進度給事故指揮官
    • 事故指揮官的主職責不是親自處理問題，而是確保問題被解決
    • 職責
      • 組織事故處理小組
      • 協調事故處理各角色
      • 決定事故處理活動的優先順序並授權
      • 了解目前的事故情形
      • 缺少的角色都由 IC 兼任
  • Communications Lead (CL): 溝通領導
    • 職責
      • 清晰且即時的溝通
      • 讓事故處理小組外的人都能知道目前事故處理情形
      • 決定事故處理情形的訊息通過哪些溝通管道發布
      • 讓事故處理小組能專注在事故處理，由溝通領導統一對外回應
      • 阻絕外部與事故處理小組的直接聯繫
  • Operations Lead (OL): 事故處理領導
    • 職責
      • 制定並執行事故處理的行動計畫
      • 與 事故指揮官 及 溝通領導保持聯繫並在事故處理期間持續協作
  • Responder: 事故處理者
    • 職責
      • 執行事故處理領導的事故處理技術方案
      • Secondary Responder 負責協助 Primary Responder
  • Users and Stakeholders

定義一個事故

事故 V.S 異常

組織必須自行定義什麼是事故(incident)? 什麼是異常(anomaly)?

  • 所有用戶無法登入服務 => 有明確的事實目前用戶無法使用服務
  • 空參考錯誤數高於閥值 => 偏離正常情形但線上服務仍可提供用戶使用

明確的定義何謂事故有助於團隊提早緩解事故並恢復服務回到可用

評估影響

當告警發生時要回答以下問題:

  • 檢測到什麼異常?
    • 服務不健康通知? API 錯誤過多告警?
  • 這個異常代表的是什麼?
    • 線上服務目前還是可用的嗎?
  • 有沒有人因為異常受到的影響?
    • 用戶還能使用線上服務嗎?

定義事故時

當你定義這是一個事故時，你就是 事故指揮官，並需要著手展開以下事情

  • 組織事故處理小組
  • 事故分類
  • 開始溝通

記得將事故發生同步給有權宣布事故的人

定義事故後

溝通領導 需要持續的對外同步事故處理情形

  • 報告服務中斷情形
  • 列出受影響範圍
  • 估計恢復的時間

一些快速緩解的方案

  • 是否有手冊記錄著類似的事故解法?
    • 考慮使用手冊的建議解法來解決問題
  • 是否有服務更版導致服務異常?
    • 退版可能是最佳解
  • 有沒有可替換的系統?
    • 與剛剛替換下來的舊版服務做熱交換?
    • 如果有 IaC 的話，用 IaC 建立一整套新的服務來替換?

持續的狀態更新

持續向用戶及利害關係人更新目前的事故處理進度，而不是直到事故處理完後才告知

  • 目前的事故具體發生原因
  • 目前的事故我們所掌握的狀況/改變了什麼狀況
  • 事故仍然持續發生中

有系統且有邏輯的解決問題

間歇姓故障可能來自多個同時發生的因素，系統性的解決方法如下

  1. 切分並減少問題空間，逐步縮小除錯範圍
  2. 逐步檢視系統，在腦袋中思考系統的這些部分在做什麼，以及是否會導致問題
  3. 如果發現系統的某個部分不太對但不是很確定，就增加監控或 log 來證明。如果找到問題則解決之

反覆的縮小除錯範圍並排除或解決發現到的問題來解決整個問題

事故緩解的回應

當事故已止血且緩解後，我們需要通知利害關係人事故的狀態。

溝通領導 需要通知利害關係人

常用術語:

  • 事故已排除
  • 服務已恢復
  • 用戶不再受影響

排除故障的根本原因

在事故緩解過後，事故處理小組的任務必須尋找發生事故的根本原因才能排除故障。
通常來說梳理事件緩解的方法有助於我們了解造成事故的根本原因。

團隊除了持續調查根本原因外，也要持續記錄問題原因並向團隊和利害關係人通報追蹤進度

找到並解決事故的根本原因

事故的根本原因通常是來自於

  1. 系統
  2. 流程
  3. 行為

找到問題後在團隊分享經驗可以共同學習避免重蹈覆轍

解決事故

當確定根本問題後

  1. 開始實作永久修復
  2. 紀錄解決方案
  3. 同步給溝通領導

在實作完永久修復後要計畫移除臨時緩解的措施

事故解決後

溝通領導通知利害關係人及用戶

  • 根本問題分析已完成
  • 事故已得到解決
  • 相關紀錄已留下事故報告

事故報告

一個事後檢討紀錄會包含以下項目

  • 事件發生的 時間 以及 細節
  • 為了減緩或解決事件採取的 行動
  • 事件的 影響範圍
  • 事件發生的 根本原因
  • 防止再次發生事件的 後續行動

一個良好的事後檢討可以幫助團隊

  • 確保團隊了解完整的錯誤起因
    • 一個錯誤事件的發生可能是多個根本原因組合在一起造成的，妥善的執行事後檢討以及紀錄內容能有效的釐清事件根本起因以及其解法，並將知識傳遞給整個團隊。
  • 定義有效步驟預防再次發生
    • 在事件發生的當下，團隊可能做了一些處理，但在短期或長期這個事件可能會再次發生，團隊需要防止錯誤事件再次發生，並為此預防工作做出排序。
  • 減少錯誤發生的可能性
    • 每次的錯誤事件對團隊來說都是一種壓力，從錯誤中學習，從而避免錯誤的發生及再次發生，讓團隊可以花費更多的力氣在改進系統上。
  • 避免複雜性
    • 大部分發生錯誤事件時我們很常用的是短解，這樣的修復就像是衣服上的補丁一般，如果沒有良好的事後分析並做長解，隨著系統不斷的功能疊代，修復工作可能會更加困難，系統的可維護性會大大降低。
  • 從錯誤中學習
    • 每次的錯誤都是一個良好的學習機會，透過事後檢討的共享，這個錯誤為整個組織提供了額外價值。專注於不責怪的檢討讓團隊完全專注於回顧事件並防止事件再次發生。
