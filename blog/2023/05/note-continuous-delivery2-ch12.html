高頻發佈是一種趨勢

世界領先的網際網路公司都在以頻繁發佈的模式更新他的的軟體產品

獲益與成本共存

好處

  1. 有多機會與真實用戶互動，進而快速決定或調整自己產品前進方向
  2. 每次變更規模較小，降低部署風險
  3. 單次部署成本降低且趨於穩定
  4. 出現問題易定位易修復

成本

  1. 發佈的成本因次數提高而總數提高

降低發佈風險的方法

藍綠部署

  1. 兩套一樣的執行環境，一套是生產環境，另一套是預生產環境
  2. 部署到預生產環境環境，確認沒問題後將流量導到預生產環境環境
  3. 此時預生產環境和生產環境職責交換
  4. 由於資料庫成本高，大部分情況會是使用相同的資料庫來達到藍綠部署
  5. 須注意新舊版本之間的事務處理

滾動部署

  1. 多個執行環境逐步更新直到全部更新完成
  2. 成本會低於藍綠部署
  3. 無法做到像藍綠部署透過流量切換快速退版

金絲雀發佈與灰度發佈

  • 金絲雀發佈

讓一小部分用戶先使用新版以提前發現軟體存在的問題，進而不免讓更多用戶受到傷害的發佈方式，因為僅有一小部分用戶使用，影響範圍較小。

  • 灰度發佈

將發佈分成不同階段，每個階段的用戶數量逐級增加。逐步擴展用戶數量到 100 %

暗部署

在功能或特性正式發佈之前，將第一個版本部署到生產環境，
開發團隊可以在功能正式對外前在生產環境先行測試

高頻發佈支撐技術

某個功能比較複雜，無法在兩次之間完成開發時的解決方法?

  1. 拆分功能:
    將功能分解為更小的功能，在一個週期內做交付
  2. 先後在前:
    先實作伺服器端功能，再實作用戶介面
  3. 功能開關技術(Feature toggle):
    1. 商業軟體根據授權提供功能
    2. 隔離: 未完成功能隔離在執行路徑
    3. 快速止血: 一旦環境出問題，直接找對應功能開關選項，將其設置為關閉

資料轉移技術

  1. 只增不刪
    透過資料庫加欄位來讓應用程式可以修改較小，且資料庫的遷移工作較低

    比方說有一資料表如下

    城市地區具體地址
    台北市中山區南京東路三段 377 號

    如果實際業務場景都只需要完整地址，則可以增加一個欄位來存放，並修改應用程式

    城市地區具體地址完整地址
    台北市中山區南京東路三段 377 號台北市中山 區南驚動路三段 377 號

  2. 資料移轉 有幾種可能用 方法 1 是無法適用的，像是

    1. 資料庫儲存系統更換
    2. 原有系統拆分
    3. 表單資料量過大

    則可以用資料轉移來處理，通常會有 5 個步驟:

    1. 為資料庫結構增加一個新版本
    2. 同時向兩個版本的結構寫入資料
    3. 將原本的歷史資料以背景寫入的方式回填到新版本結構
    4. 新舊版本資料比對進行確認
    5. 確認無誤後，應用程式改用新版本結構

抽象分支方法

不建立真實分支，透過設計手段，將大的重構專案分解成很多個小的程式碼變更步驟，逐步完成重大的程式碼架構調整。

  1. 找到要替換的部分程式碼
  2. 在這部分程式碼與其他程式碼之間插入一段隔離程式碼，新舊程式碼之間都透過隔離的程式碼進行互動
  3. 實作新程式碼逐步替代舊程式碼
  4. 替代完成後移除舊程式碼

好處

  1. 重構的同時也能交付業務功能需求
  2. 可以逐步驗證架構調整的方向和正確性
  3. 如果遇到緊急的情況，很容易暫停，且不浪費之前的工作量
  4. 能夠強化團隊的合作性
  5. 使軟體架構更模組化，更容易維護

成本

  1. 修改的時間周期可能會拉長
  2. 由於是疊代完成，整體工作量比一次性完成的情形要大

升級替代還原

影響發佈的因素

  1. 增量發佈帶來的獲益和可能性
  2. 每次發佈或部署的操作執行成本有多高
  3. 出現問題的概率與由這些問題帶來的成本有多少
  4. 維護同一軟體的眾多不同版本帶來的成本
  5. 高頻發佈模式對工程師的技能要求
  6. 支撐這種高頻發佈所需要的基礎工具設施與流程完整性
  7. 組織對這種高頻發佈的態度及文化取捨

小結

  1. 降低發佈風險的手段
    1. 開關技術
    2. 資料庫遷移技術
    3. 藍綠部署
    4. 金絲雀(灰度)發佈
    5. 抽象分支
    6. 暗部署
    7. 小步完整的程式碼提交策略
  2. 如果無法高頻的對外發佈軟體，在預生產環境進行發佈和部署提早得到相關品質回饋也是一個方案
  3. 將每次發佈的平均成本降到足夠低，那麼將會直接改變團隊的產品研發流程
