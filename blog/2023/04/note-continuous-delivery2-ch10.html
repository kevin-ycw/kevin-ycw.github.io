自動化測試的定位

測試領域的四類基本活動

  • 問題認知
  • 分析
    • 測試分析和設計: 對業務問題的認知，分析並設計能夠驗證是否成功解決業務問題的方式和方法
  • 執行
    • 執行由測試分析環節產生的測試案例，得到測試結果或資料
    • 存在大量重複性的勞動
  • 決策
    • 根據測試結果做出下一步行動判斷

敏捷測試

  • 業務導向專家: 與業務專家無障礙溝通
  • 技術導向人員: 容易與技術人員達成共識
  • 支援程式設計是指他的 ˋ 第一目標，是為了幫助產品研發團隊自己檢察功能需求是否開發完成
  • 評判專案: 用於找出產品是否有缺陷

自動化測試的優勢

  • 減少失誤率: 每次都會執行相同步驟並記錄，不受人的影響
  • 節省時間和執行成本: 重複性工作可以由人來執行
  • 提升測試覆蓋度: 增加測試的深度和範圍，如可以查看應用程式的執行情況
  • 做手動無法完成的測試: 模擬上先個用戶與應用程式進行互動
  • 為開發人員提高品質回饋速度: 開發人員也可以共享自動化測試
  • 提高團隊士氣: 由自動化測試運行重複性任務，讓團隊將時間花費在更有挑戰性及更有價值之活動

自動化測試需的投入

  • 工具投入成本: 測試工具的研發投入，以及人員的培訓
  • 案例維護成本: 自動化測試的案例必須隨著軟體的功能添加調整刪除等做對應調整
  • 專業技能人員成本: 人員必須有掌握軟體設計與程式碼編寫能力
  • 設備資源的投入: 自動化測試無法完全替代手動測試，因此除了要有手動測試的測試環境，也要有位自動化測試案例的執行準備相應的測試環境

自動化測試

  • 長處: 檢查軟體系統是否按照工程師編寫程式碼的設計正確執行，在批量回歸驗證相對擅長
  • 短處: 無法觀察執行過程

手動測試

  • 長處: 核心價值是在測試過程中主動觀察學習和分析，進而可以做更具有創造性的工作，例如探索測試、用戶友好性驗證或用戶體驗改善。在一次性執行品質驗證工作時成本低於自動化測試
  • 短處: 回歸測試時成本投入會高於自動化測試

突破傳統自動化測試的困境

自動化測試的建立流程

  1. 測試分析者設計測試案例並檔案化
  2. 測試執行者執行測試案例並報告 Bug
  3. 開發人員修復 Bug
  4. 測試執行者再次執行測試直到驗證通過
  5. 測試自動化專家從測試案例檔案中，選出一些重要且變動可能性較小的測試案例
  6. 對挑選出來的重要測試案例編寫自動化腳本，並歸入自動化回歸測試案例庫

傳統自動化測試的特點

  1. 測試案例執行成本高
    1. 黑盒自動化測試通常是抹你介面操作做系統整合測試，需要為每個案例準備執行資料且要清理髒資料
  2. 自動化測試的執行頻率低: 只在軟體開發完成後做為進入測試階段的准入標準以及回歸測試使用
  3. 品質回饋滯後: 大部分測試案例是對軟體應用操作流程的回歸測試，無法覆蓋目前版本開發的新功能，且由於是對整個操作流程做測試，會隨著操作流程的不同而有不同長度的等待時間，會較晚得到回饋
  4. 測試環境準備成本高: 需要準備對應所有待測案例的資料及執行環境
  5. 測試結果可信度相對低: 可能受到機器硬體、網路狀況、案例處理流程長度等因素導致隨機的測試失敗
  6. 人員依賴性強: 依賴少數測試開發專職人員，如果功能不斷迭代，自動化測試案例的投資報酬率可能會愈來愈低

自動化測試的分層

執行時間測試類型被測對象被測範圍
長用戶驗收測試整個軟體應用大
系統整合測試支撐軟體執行的一個或多個子系統
元件測試某個子系統的內部元件
短單元測試單個元件的內部單元小

傳統自動化測試產生的測試案例類型多為用戶驗收測試和系統整合測試，測試範圍大且酖個測試案例執行時間長，開發人員通常不參與自動化測試的建設，因此元件測試及單元測試的數量會較少

持續整合的驗證回饋: 快、捷、時、信

快速: 10 分鐘內能跑完最好 便捷: 團隊中的每名工程師都可以執行自動化測試，如果不夠便捷，團隊成員就會親像於推遲回饋 即時: 一旦功能發生改變就能夠透過自動化測試案例執行來告知本次程式碼變更對軟體品質的影響 可信: 自動化測試案例執行後的結果是可以被信賴的

要達成上述四個好的實踐，就必須下層的測試數量多於上層的數量

不同類型的測試金字塔

  1. 微核架構測試金字塔
    1. 端到端自動化測試: 模擬介面操作來驅動的測試
    2. API 自動化測試: 透過 API 介面驅動的測試
    3. 框架元件與外掛間服務/介面自動化測試: 元件或服務之間的功能正確性
    4. 框架或外掛的自動化測試: 單個元件或框架本身的品質驗證
    5. 自動化單元測試: 最細粒度測試
  2. 微服務應用測試金字塔
    1. 端到端自動化測試: 確保整個堧體服務在各系統間傳遞正確的資訊
    2. 工作流測試: 啟動二個以上的微服務進行業務流程的測試，驗證多個被測的服務之間是否正常工作
    3. 契約測試: 軟體系統中各服務間互動的資料標準格式，通常指的是消費者介面與服務者介面之間的正確性
    4. 業務服務或元件測試: 單個元件的測試，一個元件指的是一組相關業務邏輯的集合
    5. 單元測試: 單元指的是人為規定的最小被測業務邏輯，像是一個方法或是一個類別，特點是沒有外部依賴，驗證速度快

自動化測試的實施策略

增加自動化測試案例的著手點

  1. 針對程式碼熱區補充自動化測試案例: 常改的功能先做
  2. 跟隨新功能開發的進度: 跟隨該發進度編寫對應的自動化測試案例，由軟體開發工程師自行負責編寫自動化測試，
  3. 從測試金字塔的中間曾向上下兩端擴展: 從契約測試或是 API 測試開始做對於沒有自動化測試案例的團隊投入產出比最高
  4. 自動化測試案例的品質比數量重要
    1. 自動化測試也是程式碼，需要維護，就會有成本，以可達成對品質保證信心的自動化測試數量為目標即可
    2. 不同層級的測試不要寫相同測試案例
    3. 在實作成本最低的測試層級上進行相應業務邏輯的測試

提高自動化測試案例的執行次數

  1. 共享自動化測試案例: 讓更多的人可以使用
  2. 良好的自動化測試特徵
    1. 案件之間相互獨立
    2. 測試案例執行結果必須穩定
    3. 執行速度要快
    4. 測試環境應統一

共享自動化測試的維護職責

測試程式碼不該是二等公民，一樣需要易於維護且避免測試程式碼的逐漸腐敗

程式碼測試覆蓋率

  • 測試覆蓋率可以當成一種參考指標，因為我們無法衡量被覆蓋的程式碼衣錠是真正經過良好驗證的程式碼
  • 自動化測試案例大多時候會由開發工程師自己負責

用戶驗收自動化測試要點

  1. 搭建分層框架
  2. 測試案例數應保持少量: 以使用者旅行地圖的方式來驗證軟體應用或服務的核心工作流程
  3. 用戶驗收測試應該驗證端到端行為而非實作細節。 e.g: 用戶驗收測試在系統登入流程驗證上僅驗證是否正確執行，而不驗證輸入資訊，因輸入資訊的驗證可以用更低成本的自動化測試案例做覆蓋

為自動化測試案例預留 API

盡量用 API 來驅動業務流程的執行

  • 用介面操作通常反應較慢
  • 元素有時候不好定位

為除錯做好準備

  • 完整的日誌檔案
  • 自動螢幕截圖
  • 出錯時現場鏡像保存

測試資料的準備

  1. 確保應用程式啟動所需的最基本資料
  2. 某一類測試案例需要的資料
  3. 某些測試案例需要的額外資料

準備方法:

  1. 編寫程式自動產生資料
  2. 錄製手動測試
  3. 生產環境的非敏感資料 copy 一份或資料隱碼
  4. 進行生產環境的自動化錄製保存並備份

其他品質檢查方法

  1. 差異批註測試方法: 比對結果，如果與基準的結果有差異則人工審核
  2. 程式碼規範檢查與程式碼靜態檢測
    1. 規範檢查: SonarQube、Checkstyle、PMD
    2. 靜態掃描: Coveerity、Klocwork
    3. 動態分析: Valgrind、Purify

小節

Lead time 愈短，交付效率愈高，愈能提升客戶滿意度。隨著交付周期提高，自動化測試就愈重要

軟體快速交付對自動化測試的 4 個基本要求

  • 快速
  • 便捷
  • 可信
  • 即時

自動化測試的實踐管理

  1. 自動化測試案例執行次數愈多，平均成本愈低，獲益就愈大
  2. 自動化測試案例之間應該盡可能互相獨立
  3. 在品質有保障的前提下，自動化測試案例的數量愈少愈好
  4. 遺留程式碼的自動化測試從程式碼熱區開始 => 常修改的程式碼
  5. 自動化測試案例從測試金字塔中間層開始補充，投入產出比最高
