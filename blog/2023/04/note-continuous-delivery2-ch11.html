將一切設置管理

軟體設置管理是一切自動化的基礎，一切自動化是持續交付部屬流水線的重要原則。

設置管理的定義

  • 軟體生命週期管理: 與版本控制和基準線管理相近
  • 環境準備和應用部屬: 與維運領域的設置管理資料庫(Configuration management database)緊密相關

設置管理的目標

  • 可追溯性: 任何人在獲得授權的前提下能夠找到該軟體的任何變更歷史
    • 任何一次軟體變更都可以回答 5W1H: 誰(Who)在什麼時間點(When)做了什麼(What)，為什麼做?(Why),以及如何做(How)
    • 軟體組織中資訊安全管理的一個重要保障手段
  • 可重現性: 任何人在獲得授權的前提下，能夠重現從過去到現在之間的任意時間點的軟體狀態
    • 協助資訊安全管理
    • 提升各角色協作效率最重要手段之一

設置管理的範圍

軟體設置管理的 3 個基本原則

  1. 一切皆有版本
    1. 軟體執行堆疊的所有層次都需要做版本管理: 操作系統層(OS)、標準軟體層(PHP 等)以及應用軟體層(元件庫 & 應用程式套件等)等
  2. 共享唯一受信源
    1. 所有團隊成員都應該透過唯一受信源的內容為基準做協作
    2. 唯一受信源有三種倉庫
      1. 需求倉庫: 版本需求描述和驗收條件
      2. 程式碼倉庫: 保存所有原始程式碼的變更歷史
      3. 軟體套件倉庫 保存部屬流水線生產加工出來的軟體套件
        1. 臨時產出物倉庫
        2. 正式發佈套件倉庫
        3. 外部軟體私服庫
  3. 標準化與自動化
    • 基準線管理: 基準線是所有倉庫在某一時刻的快照
    • 標準和規範: 分支策略、分支命名、分支 Tag name、產品特性名稱以及存放位置
    • 有規範後較容易將俟常事務性操作進行自動化
    • 規範化和標準化並不等於僵化

軟體套件的版本管理

套件管理反模式

將相依的軟體套件直接與原始碼一同提交到版控中的問題

  1. 以二進位形式對於多數版控系統的傳輸是低效的
  2. 容易有很多重複的二進位軟體套件
  3. 相依的軟體套件不好進行版本控管

集中式套件管理服務

將所有的軟體套件統一存放，並對企業內部團隊提供穩定的查詢、獲取和申請等服務

  • 統一存放
  • 一致性
  • 安全放心: 統一安全掃描及審查
  • 下載速度快

軟體套件的元資訊

  • 其自身唯一標籤資訊
    • 容易理解及記憶
    • 方便查詢索引
  • 來源資訊
    • 說明對應的程式碼在哪裡
  • 依賴關係
    1. 建置時依賴: 建置階段就必須存在
    2. 測試時依賴: 測試框架
    3. 執行時依賴: e.g: c++ so 檔

套件依賴管理

  1. 使用套件可以快速搭建服務，具有重用和便利性
  2. 套件依賴很難消除，但可以透過管理辦法讓依賴關係的負面影響盡可能少
  3. 套件依賴管理的 3 種方法
    1. 顯式宣告依賴: 透過事先約束的描述方式顯式紀錄在檔案上
    2. 自動管理依賴: 在有顯式宣告依賴後，可以透過工具達成自動管理依賴
    3. 減少複雜依賴: 軟體套件倉庫管理系統最好提供檢查套件依賴關係的功能並讓使用者可以查看其中的依賴狀況甚至是依賴衝突及循環依賴等問題
      • 常見的三種套件依賴問題
        1. 依賴過多或鍊條過長
        2. 依賴衝突
        3. 循環依賴

環境基礎設施管理

環境準備的四種狀態

  1. 蠻荒 = 人腦 + 手動
    1. 開發人員自己搞定所以軟體部屬相關問題
    2. 所有與環境準備相關的知識都存在於個別骨幹開發人員的頭腦中。
  2. 規範化 = 檔案 + 私有腳本
    1. 開發人員將上線操作說明交給維運人員
    2. 規範化的上線部署流程
    3. 利用私有化腳本提升個人效率
    4. 這個階段的挑戰
      1. 透過人來維護流程，常常會有遺漏
      2. 透過郵件來追蹤檔案，搜尋不方便
      3. 審查工作量大，且因為手動工作量大，會有人繞過流程
      4. 自動化腳本不規範統一，容易出錯並導致部署過程中斷
  3. 標準化 = 辦公自動化
    • 更多的維運工程師 & 產生維運自動化平台
    • 好處
      1. 流程在平台上固化，開發團隊及維運團隊的協作在同一個平台上，協作流程受控
      2. 部分內容標準統一
      3. 可以部份重用，減少工作量
      4. 審查工作比較容易，都保存在平台上
    • 挑戰
      1. 複雜操作依然需要人工參與
      2. 兩次上線部署差異對比仍舊相對困難，每次上線清單都是獨立副本的情況下必須人工肉眼比對搜尋
  4. 自動化 = 受控式自動化腳本
    1. 平台管理自動化腳本，分為二種
      1. 操作程序式: 模擬手動執行步驟
        1. 符合原有思考習慣
        2. 靈活，只要手動操作辦的到，程序式腳本就能辦到
        3. 執行腳本前需要知道系統原有狀態
        4. 多次執行此種腳本可能使系統處於一種未知狀態
      2. 狀態宣告式: 腳本中指定環境的目標狀態
        1. 明確知道執行腳本後的狀態
        2. 容易版控
        3. 學習成本較高，通常此類腳本是 DSL (Domain-specific language)
        4. 腳本很大的時候可能除錯困難
        5. 執行方式可細分為 pull 跟 push
          1. pull 特點是目標機器上要有 Agent，指令會發送給 Agent 執行
          2. push 特點是不需要 Agent,但需要給予控制端對應的目標機器權限

環境基礎設施即程式碼

好處

  1. 無論哪類環境出了問題，都可以快速自動化建置出一個全新環境
  2. 只要獲得授權，任何人都可以完成這項任務
  3. 任何對環境的修改都可以被記錄和審查
  4. 對不同環境來說，只要將其程式碼掃描進行比對，就可以聊解他們的差異，無須登錄到實際主機上查看

環境基本設施的版本管理應該包含

  1. 作業系統名稱、版本號碼、修補程式版本號碼以及系統即設置資訊
  2. 軟體套件所依賴的中間件以及第三方軟體的對應版本號
  3. 會互動到的外部服務及其版本號碼

軟體設置項目的管理

Binary 與設置項目的分離

Binary 需要編譯程式碼後才會產生，與設置項目分離可以避免要改設置就要重新編譯

設置資訊的版本管理

設置項目的類別

  • 環境設置項目(enviroment configuration)
    • 網域名
    • 通訊埠號
  • 應用設置項目(application configuration)
    • 帳戶密碼
    • 初始分配記憶體大小
  • 業務設置項目(business configuration)
    • 功能特性開關
    • 商品定價策略調整
建置時(靜態設置)部署時(動態設置)執行時(動態設置)
環境類資料來源地址 / 註冊中心地址 / 設置中心地址
應用類帳號密碼 token 許可證快取大小
業務類String 的 application.xml折扣策略、紅包開關

在建置時避免注入環境資訊

  • 滿足 一套程序多環境部署，進而達成 所有環境中所有測試 binary 程序前後一致

設置項目的儲存組織方式

  • 資料庫
  • 設置中心服務
    • ZooKeeper
    • etcd

設置飄移

  • 設置發生未預期的修改
    • 通常都是人為臨時修改引起的
    • 不可變基礎設設施是一種解決方法

不可變基礎設施與雲端應用

滿足不可變基礎設師的三個要求

  1. 系統執行環境準備以自動化方式完成
  2. 一旦準備完成，該基礎設施任一層次均不得更改
  3. 以替代來取代對原系統進行環境更改

不可變基礎設施的實作

  1. 物理機/虛擬機的鏡像技術
    1. 使用建立好的鏡像檔產生出新的機器(e.g: vargeant up)
  2. Docker 容器化技術
    1. 相對於 物理機/虛擬機的鏡像技術 更為輕量
  3. 雲原生應用程式

優勢與挑戰

優勢

  1. 簡化維運工作
  2. 部屬流程來自檔案
  3. 持續部署不停機
  4. 減少錯誤和威脅
  5. 多類環境基礎設施一致
  6. 杜絕了 設置飄移
  7. 被測試過即是被使用
    1. 服務執行環境的環境相依設定也在驗證品質階段被驗證了

挑戰

  1. 不可變基礎設施的建立初期成本較高
  2. 生產環境突發問題修復時間可能稍長
  3. 鏡像檔分發較消耗網路資源
  4. 有狀態儲存的軟體服務不容易直接替換

資料版本的管理

  1. 資料庫結構變更
    1. 一份完整個 Schemaa 以及增量的 patch
    2. 工具舉例: flyaway 或 liquibase
  2. 資料檔案
    1. binary 檔案可以透過參照的方式來與原始程式碼做關聯進而達到統一管理(e.g: url)

小結

軟體設置管理的 3 個核心原則

  1. 對一切進行版本管理
  2. 共享唯一受信源
  3. 標準化與自動化

用下面 5 個問題來驗證檢查是否對一切做了版本管理

  1. 產品原始碼 和 測試程式碼 是否放入了 版本控制系統
  2. 軟體應用的設置資訊是否放入了版本控制系統
  3. 各類環境的系統設置是否放入的版本控制系統
  4. 自動化的建置和部署腳本是否放入了版本控制系統
  5. 軟體套件是否進行了版本管理

檢查軟體 設置管理是否做的夠好

  1. 一鍵化的從原始碼倉庫檢出(checkout)並建置出完整的軟體套件
  2. 在沒有他人的幫助下，任何團隊成員都能一鍵化自動搭建出一套應用軟體系統來體驗產品新功能
